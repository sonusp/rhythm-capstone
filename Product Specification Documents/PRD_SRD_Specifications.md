# Google Rhythm: Comprehensive System Specifications (PRD & SRD)

This document serves as the absolute source of truth for the **Google Rhythm** project. It contains the Product Requirements Document (PRD), System Requirements Document (SRD), and the complete technical architecture of the AI and security engines.

---

## Part 1: Product Requirements Document (PRD)

### 1.1 Product Vision
Google Rhythm is a highly aesthetic, privacy-first, AI-powered women's health and cycle tracking companion. It moves beyond standard calendar logging by utilizing real-time Voice AI, clinical LLM symptom extraction, and empathetic conversational interactions, all while maintaining a 100% offline, zero-knowledge architecture.

### 1.2 Target Audience
Individuals looking for a deeply personal, intelligent, and secure way to track menstrual cycles, fertility, perimenopause, or general hormonal symptoms without sacrificing their personal data to cloud servers.

### 1.3 Core Product Features
1.  **Multi-Stage Lifecycle Modes:** The UI and AI dynamically adapt based on the user's selected mode (Cycle Tracking, Try to Conceive, Pregnancy/Postpartum, Perimenopause, Childfree).
2.  **Voice-First Logging:** Users can log complex physical and emotional symptoms simply by speaking naturally to the app.
3.  **Conversational "Glowing Orb" AI (In Progress):** A real-time, 2-way Voice Companion that listens to the user, empathetically replies out loud via TTS, and silently logs clinical data in the background.
4.  **Zero-Knowledge Security:** All data lives exclusively on the user's device. Cloud backups are optionally available via the user's personal Google Drive, or via local AES-256 encrypted Vault exports.
5.  **Predictive Insights:** The app analyzes recent logs to forecast future symptoms (e.g., predicting fatigue based on historical hormonal shifts).

## Part 2: System Requirements Document (SRD)

### 2.1 Technology Stack
*   **Frontend Framework:** React (Vite)
*   **Styling:** TailwindCSS, Lucide React (Icons)
*   **Local Database:** Dexie.js (IndexedDB wrapper)
*   **Cryptography:** Native `window.crypto.subtle` (Web Crypto API)
*   **Deployment:** Vercel

### 2.2 Security Architecture (The Rhythm Vault)
The app implements military-grade local encryption to ensure absolute privacy:
*   **Key Derivation:** User passwords are put through `PBKDF2` with 100,000 iterations and a random salt to generate a 256-bit key.
*   **Encryption Algorithm:** The entire IndexedDB JSON payload is encrypted using `AES-256-GCM` with a random Initialization Vector (IV).
*   **Export Mechanics:** The ciphertext, salt, and IV are concatenated and encoded safely into Base64 (using a chunked loop to prevent call stack overflow), resulting in a `.rhythm` offline backup file.

## Part 3: AI Engine Architectures

To ensure 100% uptime, the app employs complex "Waterfall" fallback structures for all its AI components. If a primary provider goes down or hits a rate limit, the system instantly and silently routes to the next provider.

### 3.1 The "Ear": STT (Speech-to-Text) Waterfall
When the user speaks into the microphone, the audio Blob is routed through this hierarchy:
1.  **Groq Whisper (`whisper-large-v3-turbo`)** - Primary engine. OpenAI's model running on lightning-fast Groq LPU hardware.
2.  **Deepgram Nova 2 (`nova-2-medical`)** - 1st Fallback. Highly accurate medical terminology model.
3.  **AssemblyAI (`v2/transcript`)** - Ultimate Fallback. Uses a 3-step asynchronous polling mechanism as a deep safety net.

### 3.2 The "Brain": LLM (Symptom Extraction & Chat) Waterfall
When text needs to be analyzed for clinical JSON extraction or conversational responses, it routes through this hierarchy:
1.  **NVIDIA NIM (`meta/llama-3.1-70b-instruct`)** - Primary engine.
2.  **Groq (`llama-3.3-70b-versatile`)** - 1st Fallback. Lightning-fast identical model.
3.  **OpenRouter (`meta-llama/llama-3.1-8b-instruct:free`)** - Ultimate Fallback. Free, deep network safety net.

### 3.3 The "Voice": TTS (Text-to-Speech) Architecture
*(Phase 1 Implemented, UI Integration Pending)*
*   **Primary Engine:** Deepgram Aura (`aura-asteria-en`).
*   **Mechanism:** Takes the conversational text generated by the LLM Brain and streams it back as a lifelike female audio file for the UI's `<audio>` engine to play instantly.

## Part 4: Data Schemas

### 4.1 Voice Extraction JSON Schema
When the LLM Brain extracts data from a voice log, it is mathematically forced to return this exact schema:
```json
{
  "is_health_related": boolean,
  "target_date": "YYYY-MM-DD",
  "flow": "Spotting, Light, Medium, Heavy, or None",
  "symptoms": ["Array of physical symptoms"],
  "primary_emotion": "Core feeling (e.g., Anxious, Exhausted)",
  "emotional_analysis": "1-sentence psychological breakdown",
  "clinical_flag": boolean
}
```

### 4.2 Llama Conversational Guardrails
When generating a response for the Text-to-Speech engine, the LLM is restricted by these rules:
1.  Keep it incredibly brief (exactly 1 to 2 short sentences).
2.  Be comforting, conversational, and highly empathetic. Do not sound robotic.
3.  Validate feelings/symptoms based on current Cycle Day and Lifecycle Mode.
4.  No markdown, emojis, or lists (pure spoken text only).
